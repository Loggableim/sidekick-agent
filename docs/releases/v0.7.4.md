# Sidekick v0.7.4 — Windows Python Provisioning Fix

**Tag:** `v0.7.4`
**Datum:** 2026-06-08

---

## Ziel des Releases

Frischer Windows-Installer mit `uv` aber ohne Python soll nicht mehr mit
„No usable Python 3.10+ found" abbrechen, sondern `uv venv --python` als
primären Pfad nutzen — Python-Download und Venv-Erstellung in einem Schritt.

## Änderungen

### install.ps1: `Ensure-Venv` statt `Test-Python` + `Install-Venv`

**Problem:**
Auf sauberem Windows mit `uv` aber ohne Python brach der Installer mit
Exit 2 ab, weil der alte Code `uv python install`-Progress-Output (stderr)
nicht korrekt behandelte — nur `$LASTEXITCODE` ist fatal, nicht stderr-Text.

**Lösung:**
`Test-Python` und `Install-Venv` durch eine einzige Funktion `Ensure-Venv`
ersetzt (5-stufige Fallback-Strategie):

| Stufe | Pfad | Beschreibung |
|-------|------|-------------|
| 1 | **Existierendes venv** | Wenn `.venv\Scripts\python.exe` existiert, direkt nutzen |
| 2 | **`uv venv --python 3.11`** | **Primärer Pfad** — lädt Python + erstellt venv in einem Befehl |
| 3 | **`uv python install 3.11` + venv** | Fallback mit einmaligem Retry |
| 4 | **System-Python scannen** | PATH (ohne Store-Alias), `%LOCALAPPDATA%`, `%ProgramFiles%` |
| 5 | **Exit 2** | Klare Fehlermeldung + Recovery-Kommandos |

**Weitere Fixes:**
- `$LASTEXITCODE` ist das einzige Kriterium für Erfolg/Fail — stderr-Output
  wird nur ins Log geschrieben
- Vollständige Log-Ausgabe aller uv-Aufrufe (exit code, output, retries)
- Ein Retry bei `uv python install`-Fehlschlag
- Kein bare `python` mehr im gesamten Installer
- Venv wird im `$InstallDir\.venv` erstellt (unter `%LOCALAPPDATA%`)

### Troubleshooting aktualisiert

- `uv venv --python 3.11` als Recovery-Pfad dokumentiert
- Microsoft Store Alias weiterhin als Fehlerquelle erwähnt

## Smoke/Validierung

## Bekannte Einschränkungen

- System-Python-Scan findet nur Python 3.10+ (nicht 3.9 oder älter)
- `uv venv` kann auf Systemen ohne Internet keine Python-Downloads durchführen
  (dann Exit 2 mit Recovery-Hinweisen)
