# Sidekick v0.7.4 — Windows Python Provisioning Fix

**Tag:** `v0.7.4`
**Datum:** 2026-06-08

---

## Ziel des Releases

Frischer Windows-Installer mit `uv` aber ohne Python installiert Sidekick
erfolgreich. Der Installer nutzt `uv` zur Python-Provisionierung, ohne dass
Progress-Ausgaben auf stderr fälschlich als fatale Fehler interpretiert werden.

## Fehlerursache

Drei kumulative Probleme:

1. **`$ErrorActionPreference = "Stop"` (Zeile 23):** Macht PowerShell aus
   *jedem* stderr-Output externer Programme einen terminierenden Error.
   `uv venv --python 3.11` und `uv python install 3.11` schreiben
   Download-Progress ("Downloading cpython-3.11...") auf stderr.

2. **`$script:UvCmd = "uv"` (Zeile 148):** Ohne vollständigen Pfad kann
   `Start-Process` die Binary nicht finden — `2>$null` entgeht dem Problem
   nicht, weil `Start-Process` einen anderen Command-Resolver nutzt als `&`.

3. **Fehlendes `$ErrorActionPreference = "Continue"` in `Invoke-External`:**
   Selbst mit `Start-Process` und RedirectStandardError kann PowerShell 5.1
   stderr noch als terminierenden Error behandeln.

## Änderungen

### install.ps1: Drei Fixes

| Fix | Wo | Beschreibung |
|-----|----|--------------|
| full path for uv | `Install-Uv` Zeile 148 | `$script:UvCmd = (Get-Command uv).Source` |
| EAP override | `Invoke-External` | `$ErrorActionPreference = "Continue"` temporär |
| Quote cleanup | `Ensure-Venv` | `\$VenvPath` statt `\`"\$VenvPath\`"` |

### `Invoke-External` (Start-Process mit isoliertem stderr)

Der Helper nutzt jetzt:
- `Start-Process -RedirectStandardOutput/Error` → temp files
- `$ErrorActionPreference = "Continue"` während des externen Aufrufs
- Exit-Code-Auswertung via `$proc.ExitCode`
- Vollständiger stderr/stdout im Log
