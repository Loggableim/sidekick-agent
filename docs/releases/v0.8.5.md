# v0.8.5 — WebUI API proxy for old frontend compatibility

**Release date:** 2026-06-11

## Summary

The old WebUI frontend (`web/static/`) was designed for the stdlib HTTP server (`web/server.py` + `web/api/routes.py`) which has ~100+ API routes. When `sidekick dashboard` migrated to FastAPI (`cli/web_server.py`), the old frontend's API calls all failed with 405 Method Not Allowed because FastAPI had none of those routes.

## Root cause

The `cmd_dashboard` function auto-detects that no Node.js/npm is available and serves the old `web/static/` frontend via FastAPI. But the old frontend expects routes like `POST /api/chat/start`, `GET/POST /api/settings`, `POST /api/session/*`, and 50+ others — none of which existed in the FastAPI server.

## Fix

Added an **automatic API proxy** in `cli/web_server.py`:

- When the old frontend is detected (`web/static/index.html` served, no new Vite dashboard build), an internal stdlib HTTP server (`web.server`) is started on a random port.
- A catch-all FastAPI route `@app.api_route("/api/{path:path}")` matches any `/api/*` request that no specific FastAPI route handled and proxies it to the internal stdlib backend.
- The proxy preserves the HTTP method, path, query string, headers, and body.
- SSE streaming endpoints (`/api/chat/stream`) use `StreamingResponse` with chunked forwarding.

### What this fixes

| Bug | Before | After |
|-----|--------|-------|
| Chat send (Enter/button) | 405 — send() called dead endpoint | Proxy forwards to stdlib → chat works |
| Settings save | "Save failed" — 405 | Proxy forwards → settings persist |
| Session list/CRUD | 405 | Works via proxy |
| Terminal tools | 405 | Works via proxy |
| File operations | 405 | Works via proxy |
| All other old API routes | 405 | Works via proxy |

## Files changed

- `cli/web_server.py` — Added 168 lines: imports (`functools`, `socket`, `StreamingResponse`), stdlib backend launcher (`_find_free_port`, `_is_old_frontend`, `_ensure_stdlib_backend`), proxy functions (`_proxy_sync`, `_proxy_stream`), catch-all route `api_stdlib_proxy`.

## No regressions

- Installer: install.ps1 unchanged
- Onboarding routes from v0.8.4: still registered directly in FastAPI
- No LastBrowser references
- No user-facing Hermes regression
- FastAPI-specific dashboard routes (config, profiles, models, cron, etc.) still work directly (not proxied)
