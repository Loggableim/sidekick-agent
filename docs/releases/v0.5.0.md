## Ziel des Releases

## Änderungen

### 1. Streaming-Stabilität: Session-Persistenz im finally-Pfad

`_run_agent_streaming()` in `web/api/streaming.py` speichert Sessions zuverlässig:
- **Success:** `s.save()` vor stream_end-Event (Zeile 3868)
- **Error:** Fehler-Marker + `s.save()` im Error-Pfad (Zeile 3566–3570)  
- **Disconnect:** `_last_resort_sync_from_core()` im `finally`-Block (Zeile 4289)

Keine Änderung notwendig — die Streaming-Architektur war bereits robust.
Analyse bestätigt: `Session.save()` ist im try-Block, Error-Pfad UND finally-Block
abgesichert. Der DoD-Punkt ist durch die existierende Implementierung erfüllt.

### 2. `sidekick doctor --check-providers`

**Neues Flag:** `sidekick doctor --check-providers` / `-p`

Führt einen optionalen Online-Konnektivitätstest gegen den konfigurierten Provider durch:

| Provider | Endpunkt |
|----------|----------|
| openai | `https://api.openai.com/v1/models` |
| anthropic | `https://api.anthropic.com/v1/models` |
| openrouter | `https://openrouter.ai/api/v1/models` |
| deepseek | `https://api.deepseek.com/v1/models` |
| google/gemini | `https://generativelanguage.googleapis.com/v1beta/models` |

**Eigenschaften:**
- 5s HTTP HEAD-Timeout
- Alle Fehler sind `check_warn` (non-critical), nie `check_fail`
- Ohne Flag bleibt Doctor offline-first und schnell
- Bei unbekanntem Provider: klare Meldung „skip connectivity check"

**Dateien:**
- `cli/main.py`: `-p` / `--check-providers` argparse-Flag
- `cli/doctor.py`: `_check_providers_connectivity()` Funktion
