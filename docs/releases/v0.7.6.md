# Sidekick v0.7.6 — Windows Installer venv Path Fix

**Tag:** `v0.7.6`
**Datum:** 2026-06-08

---

## Ziel des Releases

Der Windows-Installer nutzt einen einheitlichen, kanonischen venv-Pfad
(`.venv`) für alle Schritte — Python-Provisionierung, Dependency-Install,
PATH-Setzung, Desktop-Shortcut und Gateway-Start.

## Problem

`uv venv --python 3.11` erstellte das venv im aktuellen Verzeichnis (`.venv`),
aber die Dependency-Install-Schritte erwarteten `$InstallDir\venv\Scripts\python.exe`
(hartcodiertes `venv` ohne Punkt-Präfix). Dadurch fand der Installer die
Python-Installation nicht und brach mit Exit-Code 5 ab.

## Änderungen

| Änderung | Beschreibung |
|----------|-------------|
| `$script:VenvPath` | Join-Path `$InstallDir` `.venv` — einheitlich definiert |
| `$script:PythonExe` | `$VenvPath\Scripts\python.exe` — eine Quelle der Wahrheit |
| `$script:SidekickExe` | `$VenvPath\Scripts\sidekick.exe` — für Desktop-Shortcut + Gateway |
| **6 hartcodierte Pfade** | `$InstallDir\venv\` → `$script:VenvPath\` ersetzt |
| **3 weitere Pfade** | `$InstallDir\.venv\` (bereits korrekt) → $script:... normiert |

**Betroffene Funktionen:**
- `Install-Dependencies` — `VIRTUAL_ENV`, `uv pip install`, Sidekick-Bin-Pfad
- `Set-PathVariable` — venv Scripts-Verzeichnis
- `Install-PlatformSdks` — Python-EXE für SDK-Verifikation
- `Invoke-SetupWizard` — `sidekick_cli.main setup` via venv-Python
- `Start-GatewayIfConfigured` — `sidekick.exe` aus venv
- Desktop-Shortcut — `.lnk`-Ziel

## Validierung

- `$script:VenvPath` definiert: ✅
- `$script:PythonExe` definiert: ✅
- `$script:SidekickExe` definiert: ✅
- Kein hartcodiertes `\venv\` mehr: ✅ (0 verbleibend)
- Braces/Parens balanciert: ✅
- Kein user-facing Hermes/LastBrowser: ✅

## Akzeptanz

- `uv` erzeugt `.venv` im Installationsverzeichnis
- Dependency-Install verwendet `.venv\Scripts\python.exe`
- Desktop-Shortcut zeigt auf `.venv\Scripts\sidekick.exe`
- PATH setzt `.venv\Scripts\` (nicht `venv\Scripts\`)
- Zweites Ausführen ist idempotent (Erkennung des existierenden venv)
